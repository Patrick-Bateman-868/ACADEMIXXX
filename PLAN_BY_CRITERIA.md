# План разработки Gold Students Club по критериям кейса

## Цель
Концепция и прототип цифровой платформы закрытого международного сообщества выдающихся студентов с проверенными возможностями.

---

## 3.1 Профиль студента

### Требования к данным
| Поле | Обязательность | Описание |
|------|----------------|----------|
| Имя | обязательно | Отображаемое имя |
| Университет | опционально | Название вуза |
| Страна | опционально | Страна обучения/проживания |
| Навыки (skills) | опционально | Теги или список (например, через запятую) |
| Интересы и цели | опционально | Текстовые поля |
| Ссылки | опционально | LinkedIn, GitHub, портфолио — отдельные поля или одна строка |

### План реализации
1. **Модель данных**
   - Расширить `User`: поля `name`, `university`, `country`, `skills`, `goals`, `interests`, `links` (или отдельно `linkedin_url`, `github_url`, `portfolio_url`).
   - Добавить `avatar_url` (опционально), `created_at`, `updated_at`, `status` (pending/verified/banned).

2. **Страницы и сценарии**
   - **Мой профиль** (`/profile`): форма создания/редактирования; доступ только для залогиненного пользователя.
   - **Каталог студентов** (`/profiles`): список карточек с именем, университетом, страной, навыками (превью).
   - **Публичный профиль** (`/profiles/<id>`): просмотр в read-only для других пользователей.

3. **Валидация**
   - Обязательные: имя (и email при регистрации).
   - Опционально: валидация URL для ссылок.

**Результат шага:** профиль студента с полным набором полей по п. 3.1, создание и редактирование работают, данные сохраняются.

## 3.2 Возможности (Opportunities)

### Категории
- Гранты  
- Стажировки  
- Работа / проекты  
- Менторы  
- Жильё (опционально)

### Поля возможности
| Поле | Описание |
|------|----------|
| Название | Заголовок |
| Описание | Текстовое описание |
| Требования | Список или текст |
| Дедлайн | Дата или строка |
| Источник / партнёр | Название организации или контакт |
| Статус проверки | verified / unverified |
| Категория | Одна из категорий выше |

### План реализации
1. **Модель**
   - В модели `Opportunity`: `title`, `description`, `requirements` (Text), `deadline`, `source` или `partner_name`, `verified` (Boolean), `category` (enum или строка: grant, internship, project, mentor, housing).
   - Добавить `created_at`, `created_by` (FK на User), при необходимости `location`.

2. **Интерфейс**
   - Список возможностей с карточками: категория, название, дедлайн, бейдж verified/unverified.
   - Страница одной возможности: все поля, кнопка «Подать заявку» (если реализованы заявки).

3. **Добавление/редактирование**
   - Форма добавления возможности (для авторизованных или только для админов — по решению).
   - Админ: редактирование любой возможности, переключение verified.

**Результат шага:** все категории и поля из п. 3.2 присутствуют, отображаются и (где нужно) редактируются.

## 3.3 Поиск и фильтрация

### Требуемые фильтры
- По навыкам  
- По университету  
- По стране  
- По типу возможностей  
- По дедлайну  

### План реализации

**Для возможностей**
- Фильтр по категории (тип возможности).
- Фильтр по дедлайну: «ещё открыто» (deadline >= сегодня) или выбор диапазона дат.
- Текстовый поиск по названию/описанию/требованиям.
- Чекбокс «Только верифицированные».

**Для студентов (профилей)**
- Поиск по имени.
- Фильтр по стране.
- Фильтр по университету (или поиск по подстроке).
- Фильтр по навыкам: выбор из списка или поиск по подстроке в поле skills.

**Технически**
- GET-параметры для фильтров, сохранение в URL (чтобы можно было шарить ссылку).
- Пагинация результатов.
- Для дедлайна: хранить в БД дату (Date/DateTime) или нормализованную строку для сравнения.

**Результат шага:** на страницах возможностей и профилей работают все заявленные виды поиска и фильтрации.

## 3.4 Комьюнити

### Функции
- Группы по интересам / направлениям  
- Личные сообщения или чаты  
- Интеграция с Telegram/WhatsApp или встроенное решение  
- Базовая логика модерации  

### План реализации

**Этап 1 — Группы (MVP)**
- Модель `Group`: название, описание, категория/направление, дата создания.
- Связь пользователь–группа (многие ко многим): таблица `group_members`.
- Страница «Сообщество»: список групп с описанием.
- Страница группы: описание, список участников (или счётчик). Кнопка «Вступить» / «Выйти» для залогиненных.

**Этап 2 — Сообщения**
- Вариант A: встроенные чаты  
  - Модели: `Conversation`, `Message` (sender, recipient, content, created_at).  
  - Страница «Сообщения»: список диалогов и окно переписки (простой интерфейс).
- Вариант B: интеграция с Telegram/WhatsApp  
  - Ссылки «Написать в Telegram» / «Чат в WhatsApp» в профиле или в группе (открытие в новом окне).  
  - Поле в профиле: `telegram_username`, `whatsapp_phone` (опционально).

**Этап 3 — Модерация**
- Роли: `user`, `moderator`, `admin`.
- Модератор может: скрывать посты в группах, удалять сообщения, предупреждать пользователей.
- Таблица `ModerationLog` (кто, что, когда) — опционально для прототипа.

**Результат шага:** есть группы по интересам, способ общения (встроенный или через ссылки), описанная и частично реализованная модерация.

## 3.5 Административная логика

### Функционал
- Добавление и редактирование возможностей  
- Модерация пользователей и контента  
- Верификация возможностей и участников  
- Управление доступом (открытое / закрытое комьюнити)  

### План реализации

1. **Роли и доступ**
   - В модели `User`: поле `role` (user, moderator, admin).
   - Декоратор или проверка `current_user.role == 'admin'` / `in ['admin', 'moderator']` для админ-маршрутов.

2. **Админ-панель (отдельный blueprint или префикс `/admin`)**
   - **Возможности:** список всех, кнопки «Редактировать», «Верифицировать»/«Снять верификацию», «Удалить».
   - **Пользователи:** список с фильтром по статусу; действия: «Верифицировать», «Забанить», «Сделать модератором».
   - **Контент:** при наличии групп/чатов — список последних постов/сообщений с возможностью скрыть/удалить.

3. **Управление доступом**
   - Глобальная настройка: «Сообщество открытое» / «Закрытое» (флаг в конфиге или в таблице `Settings`).
   - Закрытое: просмотр возможностей и каталога только для верифицированных (или залогиненных); главная и логин — для всех.
   - В интерфейсе: условное отображение блоков в зависимости от роли и открытости.

**Результат шага:** админ может управлять возможностями, верифицировать пользователей и контент, включена базовая логика открытого/закрытого доступа.

## 3.6 UX/UI-прототип

### Требования
- Web или mobile  
- Простой и интуитивный UX  
- Фокус на реальных пользовательских сценариях  
- Инструменты: Figma или аналог  

### План реализации

1. **Сценарии для прототипа**
   - Студент: вход → главная → каталог возможностей (с фильтрами) → карточка возможности → подача заявки.
   - Студент: регистрация → заполнение профиля → просмотр своего профиля и редактирование.
   - Студент: каталог студентов → фильтры → просмотр профиля другого студента.
   - Студент: раздел «Сообщество» → список групп → страница группы.
   - Админ: вход в админку → модерация возможностей и пользователей.

2. **Структура прототипа в Figma (или аналог)**
   - Набор экранов: главная, логин/регистрация, профиль (вид и редактирование), каталог возможностей, карточка возможности, каталог студентов, профиль студента, сообщество/группы, админ (списки и формы).
   - Компоненты: шапка, карточки, кнопки, формы, бейджи verified.
   - Золотой акцент, минималистичная типографика, единый стиль.

3. **Web-прототип (текущий проект)**
   - Адаптивная вёрстка: основные экраны корректно отображаются на планшете и телефоне.
   - Единая дизайн-система в коде (цвета, отступы, компоненты) в соответствии с макетом.

**Результат шага:** готовый кликабельный/статичный прототип в Figma (или аналоге) и согласованный с ним рабочий web-интерфейс.
